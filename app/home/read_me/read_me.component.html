API Management?
Parkninja provides RESTful web services that clients (IOS, Android etc..) can access. Each revision will be tracked accordingly,
v(major change).(minor change).(bug fix) thus resulting to something like v2.0.0, this should help avoid chaos caused by constant updates.
You can search API details by revision or environment, if you select a specific environment the revision dropdown will automatically show which revision
is currently deployed on that environment.

Error Codes - some known sites like facebook for example provides API's that always return a status 200 regardless of the result so the clients need to
parse the JSON response to determine what really happened to the request. Others make full use of HttpStatus codes which adds complexity for the clients.
Since parkninja is the best we will use the best approach, we will use HttpStatus but with moderation and sum it up to 3 scenarios.
1.) Client request is successful and fulfilled without an error. (HttpStatus.200)
2.) Client passed invalid request. (HttpStatus.400)
3.) Parkninja server encountered an error while fulfilling the request. Both handled and unhandled exceptions (HttpStatus.500).

If 500 error is encountered 'errorCode' and 'message' field contains the details.
System Logs?
1.) API Error - contains request failures logs from exposed API's. Format is 'error Code', 'message', 'deviceId'(who made the request).
2.) Paynamics Transactions - logs paynamics calls, xml being sent to paynamics, xml being received from paynamics.
3.) Catalina.out - tomcat server log file contains server startup up process.
4.) ParkninjaCore Error - Parkninja-core application error logs.
5.) ParkwiseCore Error - Parkwise-core error logs.

Accounts?
You can search through all the accounts registered in Parkninja just enter the email address and hit enter.
This will show all the account information, tickets, parking history etc. Developers can deactivate test accounts here.
